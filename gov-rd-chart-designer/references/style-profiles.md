# Proposal Infographic Style Profiles

Use these profiles after selecting the chart type and layout template. The profile controls visual emphasis, canvas ratio, typography, and how the proposal argument is made obvious.

## Selection Rules

- Choose exactly one style profile for each figure unless the user explicitly asks for alternatives.
- Style profiles do not replace T1-T6 templates. Example: a task-indicator figure may use T1 plus S3; a validation loop may use T4 plus S8.
- If the user provides an accepted or preferred reference figure, match its profile first, then adapt terminology and evidence to the current project.
- A full proposal document may mix profiles, but keep color, font, and line weight consistent across the document.

## Shared Visual Tokens

- Default width: 150-160 mm in Word body text.
- Default export ratio: true content ratio, usually 16:9, 4:3, 3:2, or 1:1 only when the figure itself is circular or square. Do not center a small chart inside a square canvas.
- Core title or theme anchor inside artwork: 13-18 pt equivalent, bold, concise, not a duplicate official caption.
- Core module text: 10.5-12 pt equivalent for selected-style infographics; 9.5-10.5 pt only for dense matrices or architecture cells.
- Main line/border: 1.1-1.6 pt equivalent; support grid lines: 0.75-1.0 pt equivalent.
- Main palette: deep government blue, gray-blue, white, very light blue. Use teal, muted orange, or muted gold only for semantic emphasis.
- Grayscale survival: position, hierarchy, numbering, arrows, and labels must still carry the meaning.

## S1 Deep-Blue Spine Overview

Use for: overall framework, independent research topic positioning, project logic overview, "pain point -> core subject -> outputs" narratives.

- Structure: large short theme anchor at top; 2-4 upstream pain/problem modules; a deep-blue central spine/bar naming the project core; downstream capability/output modules below.
- Canvas: landscape 16:9 or 3:2 when inserted as a body figure.
- Visual emphasis: central spine is the strongest object; upstream and downstream modules align to it with short orthogonal connectors.
- Text: use project-specific nouns in every module, such as `动态风险场`, `多智能体协同`, `标准化数据集`, `原型系统`.
- Avoid: generic center labels like `核心平台`; small center bars; weak pale borders; scattered modules around the spine.

## S2 Layered Pyramid Principle

Use for: scientific principle, technical logic, mechanism chain, theory-to-model explanation.

- Structure: left-side layer labels plus stacked trapezoids or horizontal bands; arrows flow downward or upward through the layers.
- Canvas: portrait or landscape 4:3; fill the page width and most of the artwork height.
- Visual emphasis: layered bands show progression from scenario constraints/data evidence to domain mechanism/intelligent model/engineering validation.
- Text: layer names should be bold and short; each band contains one domain-specific mechanism phrase.
- Avoid: six or more thin layers with small text; decorative pyramids without causal wording; gradients that reduce text contrast.

## S3 Dual-Column Task-Indicator Mapping

Use for: research tasks vs assessment indicators, task-output mapping, deliverable verification.

- Structure: left column `研究任务`, right column `考核量化指标` or `验收证据`; one arrow per row; rows align exactly.
- Canvas: landscape 16:9 or 3:2; use 3-5 rows.
- Visual emphasis: right column may use stronger blue blocks to highlight quantities and deliverables.
- Text: put measurable phrases directly in the right column, such as `4类风险模型`, `原型系统1套`, `20组场景验证`, `≥3项知识产权`.
- Avoid: tiny matrix cells, vague right-column text, misaligned arrows, or equal visual weight when the indicators are the review focus.

## S4 Milestone Timeline

Use for: technical implementation stages, project schedule, phase outputs, annual/monthly milestones.

- Structure: horizontal timeline with dates or phase numbers; each node has a compact output card; optional top band for total period.
- Canvas: wide landscape 16:9 or 4:1 for narrow timeline figures.
- Visual emphasis: timeline line and dates are clear; stage output labels are readable at final print size.
- Text: each node should include date/phase, short work item, and expected output.
- Avoid: long text paragraphs, small date labels, too many nodes in one row, or ornamental timeline dots without deliverables.

## S5 Layered Architecture Stack

Use for: system architecture, platform architecture, data/model/service/application layering.

- Structure: 4-6 horizontal layers from governance/business/model/data/source or source/data/model/service/application; modules are aligned inside each layer.
- Canvas: portrait 4:3 or landscape 16:9 depending on layer count.
- Visual emphasis: layer labels are strong at left or top; modules fill the layer width; only adjacent-layer dependencies use connectors.
- Text: replace generic labels with domain modules, such as `无人机遥测`, `航线审批`, `风险场模型`, `冲突解脱`, `监管对接`.
- Avoid: web-like connector networks, tiny module grids, isolated architecture boxes, or generic `数据层/模型层/服务层` without domain terms.

## S6 Lifecycle Governance Loop

Use for: data flow, data governance, evidence chain, asset reuse, continuous data improvement.

- Structure: circular or octagonal lifecycle with 6-8 steps; center label names the reusable asset or evidence chain.
- Canvas: near-square only when the loop fills the artwork; otherwise use 4:3.
- Visual emphasis: loop arrows are thick enough to print; step nodes are equal and placed on a stable circular or polygonal grid.
- Text: center label must state the asset logic, such as `低空数据资产全生命周期复用`; nodes carry concrete lifecycle verbs.
- Avoid: weak thin circles, distant detached nodes, huge center box, or a small loop floating on a large canvas.

## S7 Review Matrix Enhanced

Use for: technical bottleneck matrix, evidence matrix, task-indicator-result matrix, acceptance review matrix.

- Structure: dense table with dedicated `序号` column; 3-5 content columns; body occupies most of the artwork.
- Canvas: landscape 16:9 for wider mappings or portrait 4:3 for shorter matrices.
- Visual emphasis: header row has pale blue fill; body text is centered and heavier than normal table text; row numbers sit in their own narrow column.
- Text: use short two-line cells; combine evidence items with slashes before shrinking type.
- Avoid: decorative row badges, floating numbers, large table margins, top-left text, sparse oversized cells, or row numbers outside the grid.

## S8 Field Validation Loop

Use for: demonstration validation, pilot application, scenario trial, expert review, iteration and rollout.

- Structure: large loop with 4-6 field-validation nodes plus a central theme/evidence label. A 2x2 compact loop is acceptable for formal body text.
- Canvas: 4:3 or near-square when the loop fills the canvas.
- Visual emphasis: loop direction is obvious; field scene, model/system output, human/expert confirmation, feedback optimization, and final evidence are visibly distinct.
- Text: include the validation setting and evidence names, such as `实景场景输入`, `现场落地试运行`, `人工专家复核验收`, `问题反馈迭代优化`.
- Avoid: generic closed loops, long diagonal arrows, empty center, or nodes that only say `输入/输出/反馈`.

## Profile Failure Check

Redraw or switch profile if:

- the project subject is not visible within 10 seconds;
- the artwork could fit any industry after changing a few labels;
- core labels become weaker than surrounding body text in a printed-page photo;
- a chart uses a square PNG canvas but the real diagram only occupies a small center area;
- the figure satisfies alignment rules but still looks sparse, timid, or not selected-quality.
